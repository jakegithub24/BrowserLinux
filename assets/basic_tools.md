## CLI tools that can be **safely simulated** in a browser‑based Linux terminal  

| Category | Tool | Typical use | How it can be emulated (what you need) |
|----------|------|-------------|----------------------------------------|
| **Text editors** | `nano` | Simple line‑oriented editor | Inline textarea with keybindings (Ctrl‑O, Ctrl‑X, etc.); save to the virtual file system |
| | `vim` | Modal editor for power users | JavaScript‑based vim implementation (e.g., **vimmer**, **wasm‑vim**) that works on a virtual FS |
| | `micro` | Modern terminal editor | Port the Go source to WebAssembly or mimic its core commands |
| **Process monitors** | `top` / `htop` | Real‑time view of processes | Fake process table generated by the emulator; update every second and display CPU/Memory usage stats derived from the sandbox |
| **File managers** | `mc` (Midnight Commander) | Two‑pane file browser | Render a split view with clickable panes; actions map to the underlying virtual FS |
| **Archivers** | `tar`, `gzip`, `bzip2`, `zip`, `unzip` | Create/extract archives | Use JS libraries (`tar-js`, `fflate`, `pako`) to read/write archive formats in memory |
| **Network utilities** | `ping`, `traceroute`, `curl`, `wget`, `nslookup`, `dig` | Test connectivity, fetch data | - **curl/wget** can perform real HTTP requests from the browser (subject to CORS). <br> - **ping/traceroute** can be mocked by returning static latency values. <br> - **nslookup/dig** can call public DNS over HTTPS APIs. |
| **System info** | `uname`, `hostname`, `date`, `cal`, `uptime` | Show kernel, time, calendar | Directly query the browser’s environment (Date API) and return static strings for kernel info. |
| **Resource stats** | `df`, `du`, `free`, `lsblk` | Disk/Memory usage | Compute stats from the in‑memory virtual file system; report fixed total sizes. |
| **Search / filter** | `grep`, `sed`, `awk`, `cut`, `sort`, `uniq`, `wc` | Text processing pipelines | Implement as pure‑JS stream filters; many already exist as npm packages (e.g., **grep‑js**, **awk‑js**). |
| **Shell built‑ins** | `echo`, `read`, `export`, `alias`, `source` | Variable handling and scripting | Straightforward to support within the JS‑based shell engine. |
| **File system navigation** | `ls`, `cd`, `pwd`, `mkdir`, `rmdir`, `rm`, `mv`, `cp`, `ln` | Basic FS ops | Directly map to the virtual FS storage (IndexedDB, localStorage, or in‑memory). |
| **Package‑like helpers** | `apt‑search` (mock), `yum‑list` (mock) | Learn package manager syntax | Return static JSON data representing a catalog; no real installation occurs. |
| **Job control** | `bg`, `fg`, `jobs`, `sleep` | Background tasks | Simulate by spawning JS timers/Promises; `jobs` lists those timers. |
| **Logging / monitoring** | `tail -f`, `journalctl` (mock) | Follow log files | Provide virtual log files that grow over time; `tail -f` streams updates to the terminal UI. |
| **Scripting** | `bash`/`sh` (interpreter) | Run shell scripts | A lightweight JS‑based POSIX‑like interpreter (e.g., **bash.js** or **shell‑js** compiled to WebAssembly). |
| **Misc utilities** | `clear`, `reset`, `history`, `type`, `which` | Convenience commands | Pure‑JS implementations that manipulate terminal UI or look up built‑ins. |

### How to implement them  

1. **Virtual filesystem** – Use **BrowserFS** or a custom IndexedDB wrapper to store files/directories.  
2. **WebAssembly ports** – Compile small C utilities (e.g., `nano`, `vim`, `htop`) to WASM; expose a virtual FS mount point.  
3. **JS libraries** – Leverage existing npm packages for text processing (`grep-js`, `awk-js`, `pako` for gzip, `fflate` for zip).  
4. **Mock data** – For commands that require privileged info (`top`, `ps`, `df`), generate static or semi‑dynamic data structures that look realistic.  
5. **Network sandbox** – Use the browser’s `fetch` API for `curl`/`wget`; for DNS tools, call public DNS‑over‑HTTPS services.  

These tools give learners a realistic command‑line experience while staying safely sandboxed inside the browser.
